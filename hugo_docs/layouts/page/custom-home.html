GO · GIN · LANGCHAINGO

A high-concurrency agentic engine for financial workloads.

An enterprise chatbot that serves cached answers in under two seconds, deduplicates identical in-flight requests into a single LLM stream, and degrades gracefully under load.

8 concurrent leaders 50 active subscribers 5 LLM stream workers < 2s cache hit 100-deep queue
Agentic engine · request lifecycle
Entry POST /submit all requests
Tier 1 Cache fast-lane hit < 2s · bypasses semaphores
Tier 2 Priority queue FIFO · miss path
Tier 3 Leader → broadcast 1 stream, many subscribers

The cache fast-lane bypasses every semaphore. A hit returns in under two seconds; a miss falls through to the priority queue.

ARCHITECTURE

System Architecture

A decoupled, two-stage worker pipeline built for high throughput. It separates fast data preparation from slow LLM interactions and executes blocking tool calls concurrently to minimise latency.

API

API Endpoints

A submit/stream/status/cancel lifecycle, plus reaction and history endpoints.

Authentication

Protected endpoints under the /chat group require an X-API-Key header matching the configured environment variable.

Endpoints

  • POST/chat/submit(Auth) Submits a query, returns a request_id.
  • GET/chat/stream/:request_id(Public) Establishes an SSE connection to stream the response.
  • GET/chat/status/:request_id(Auth) Checks request state within its lifecycle.
  • POST/chat/cancel/:request_id(Auth) Cancels an in-progress request.
  • POST/chat/update_reaction(Auth) Updates user feedback for a response.
  • GET/chat/get_chat_history(Auth) Retrieves conversation history.

SETUP

Installation & Setup

Prerequisites

  • Go 1.24.0
  • Redis Server
  • ArangoDB
  • Docker & Docker Compose
  • A configured .env file

Running with Docker Compose (recommended)

This is the canonical way to run the entire system:

  1. Clone the repository
  2. Create your .env file from the template
  3. Launch everything:
docker-compose up --build

This single command will:

  • Build the Go application container
  • Start ArangoDB, Redis, Loki, Grafana, and Vector
  • Serve the chatbot at http://localhost:8080
  • Serve Grafana at http://localhost:3000

Manual local setup

  1. Clone the repo and cd into it
  2. Install dependencies: go mod tidy
  3. Configure .env
  4. Run services: ensure Redis, ArangoDB, and dependent microservices are running
  5. Run the app: go run main.go

Configuration

# --- Token & Output Management ---

MAX_TOTAL_TOOL_OUTPUT_TOKENS=33000  # Max combined tokens from all tools before truncation.
LONG_TEXT_TOOLS=analyze_stock       # Comma-separated list of tools to truncate first.

# --- Pipeline, Concurrency & Lifecycle Tuning ---
# These settings dictate the performance and resilience of the system. Handle them with care.
MAX_CONCURRENT_LEADERS=8            # Max concurrent UNIQUE, EXPENSIVE requests (tool selection, etc).
MAX_CONCURRENT_SUBSCRIBERS=50       # Max total active requests, including cheap followers. Higher = more concurrent users.
MAX_CONCURRENT_LLM_STREAMS=5        # Number of workers in the slow LLM-interaction pool. Keep this low to avoid hitting API rate limits.
HISTORY_ITEMS=3                     # Number of past conversation turns to include in the context.
QUEUE_SIZE=100                      # Buffer size for incoming requests.

# --- State-Aware Janitor & Timeouts ---
JANITOR_INTERVAL=2m                 # Cleanup sweep interval for stuck/abandoned requests and stale tracking data.
PROCESSING_TIMEOUT=10m              # Max time a dequeued request may run before it is force-killed. Reaps stuck jobs.
QUEUE_TIMEOUT=1h                    # Max time a request can wait in the queue before being discarded. A safety net for extreme load.
TOOL_TIMEOUT=30s                    # Max execution time for a single tool call. Prevents a slow dependency from stalling everything.
CLIENT_PICKUP_TIMEOUT=1s            # Grace period for the client to connect to the stream. Solves a race condition for fast/cached responses. MUST be short.
SUBMIT_TIMEOUT=2s                   # Max wait for a slot in the request queue before rejecting. Prevents callers hanging on a full system.

Advanced concurrency & integration testing

These scripts and tests validate the system's core concurrency and resilience models under high-contention scenarios.

1. Cache Fast-Lane integration (cache_fastlane_integration_test.go)
Critical This is the critical test suite that proves the cache fast-lane bypasses all semaphores and resource constraints. 
# Run complete cache fast-lane integration tests
go clean -testcache && go test -v -run TestCacheFastLane

Critical tests:

  • TestCacheFastLane_BasicFlow: End-to-end cache population and retrieval validation.
  • TestCacheFastLane_BypassesSemaphores: Most important — proves cache requests complete instantly even when all semaphores are blocked.
  • TestCacheFastLane_ThreeTierPriority: Validates the complete three-tier system (Cache → Follower → Leader).
  • TestCacheFastLane_PerformanceComparison: Quantifies performance improvement (>1.5x speedup required).
  • TestCacheFastLane_ConcurrentCacheHits: High concurrency validation (5+ simultaneous cache requests).

Expected results:

  • Cache responses complete in <2 seconds.
  • Cache bypasses blocked semaphores completely.
  • Performance improvement >50% vs normal processing.
  • 100% response consistency between cached and original.
2. Manager cache integration (cache_manager_integration_test.go)

Validates cache fast-lane integration at the manager level:

go clean -testcache && go test -v -run TestManagerCacheFastLane

Integration tests:

  • TestManagerCacheFastLane_Integration: Manager routing through the cache fast-lane.
  • TestManagerCacheFastLane_WorkerDistribution: Dedicated cache workers vs normal workers.
  • TestManagerCacheFastLane_StateManagement: Request lifecycle and cleanup.
  • TestManagerCacheFastLane_ErrorHandling: Graceful degradation when Redis is unavailable.
  • TestManagerCacheFastLane_StressTest: 50+ concurrent cache requests (>90% success rate required).
3. Complete Cache Fast-Lane test suite

Execute the comprehensive cache test script:

chmod +x run_cache_tests.sh
./run_cache_tests.sh

This script runs:

  • Cache package unit tests with race detection
  • All cache fast-lane integration tests
  • Manager integration tests with cache
  • Performance benchmarks
  • Stress testing (optional, skipped in CI)
4. General throughput (test_concurrency)

This script validates that the server can handle many different requests in parallel, testing the capacity of the worker pools.

  1. Run the test: 
    # Usage: go run ./cmd/test_concurrency -n <num_requests> -w <server_workers>
go run ./cmd/test_concurrency -n 20 -w 10
  2. Analyze results: the script generates an Excel file like test_results/parallel_test_r20_w10.xlsx. Check that all requests completed and the Time-To-First-Token (TTF) values are reasonable.
5. Single-Flight protection (test_single_flight)

This script validates thundering-herd protection by sending many identical requests simultaneously.

  1. (Optional but recommended) Clear the relevant Redis cache key before testing to ensure you are not just hitting the cache.
  2. Run the test: 
    # Usage: go run ./cmd/test_single_flight -n <num_requests>
go run ./cmd/test_single_flight -n 10
    Verification: the proof is in the server logs. Look for one "leader" request performing the full preparation, while all other "follower" requests share its result, indicated by the This request is a FOLLOWER log message.
6. End-to-end cancellation (stream_test.go)

This integration suite validates that a request can be terminated at any point in its lifecycle and that the system reclaims resources gracefully. This is crucial for preventing resource leaks from abandoned requests.

The test simulates every cancellation scenario:

  • Happy path: a standard request is submitted and streams to completion without interruption.
  • Pre-emptive cancellation: a request is cancelled immediately after submission, before the client connects to the stream. Verifies a cancelled status is correctly returned.
  • Mid-stream cancellation: a request is cancelled after streaming has begun, proving that the context is correctly propagated to terminate the in-progress LLM call.
  • Post-completion cancellation: a request is cancelled after it has already finished, verifying the server correctly returns a not_found status, proving proper state cleanup.
  1. Run the test: 
    # This command specifically runs all tests prefixed with "TestStream"
go clean -testcache && go test -v -run ^TestStream .
  2. Analyze results: observe the test logs. A successful run shows each scenario passing, confirming the system handles all cancellation states correctly.
7. Real-time request status (get_status_test.go)

This integration suite validates the reliability and accuracy of the GET /chat/status/:request_id endpoint, which is essential for providing real-time feedback to the user.

Key validations:

  • Active states: verifies correct status reporting for requests that are queued, choosing_tools, calling_tools, and formulating_answer.
  • Database fallback: confirms that the status for completed and cancelled requests can be retrieved from the database after they are no longer in active memory.
  • Progress tracking: ensures the detailed progress object (completion ratio, selected tools, etc.) is accurately reported throughout the lifecycle.
  • Error handling: checks for correct 404 Not Found responses for non-existent requests.
  1. Run the test: 
    # This command specifically runs all tests prefixed with "TestStatusEndpoint"
go clean -testcache && go test -v -run ^TestStatusEndpoint .
  2. Analyze results: a successful run confirms that the status endpoint provides a reliable and complete view of a request's journey from submission to completion.
8. Priority dispatch & FIFO integrity (priority_dispatch_test.go)

This is the most advanced concurrency test, designed to rigorously validate the intelligent dispatching logic that maximises throughput under load. It proves that the system correctly prioritises cheap "follower" requests over expensive, blocked "leader" requests, while still maintaining fairness (FIFO) for the waiting leaders.

The test constructs a precise, high-contention scenario:

  1. Saturates leader slots: deliberately fills all MaxConcurrentLeaders slots with unique requests that are paused mid-stream, creating a bottleneck.
  2. Queues new leaders: submits additional unique requests that are forced to wait in the queue because no leader slots are available.
  3. Introduces followers: submits follower requests (which match the in-progress leaders) after the new leaders have been queued.
  4. Verifies correct behaviour: the test passes only if two conditions are met:
    • Follower fast-tracking: the follower requests must jump the queue and be processed immediately, bypassing the waiting leaders.
    • Leader FIFO integrity: when a leader slot is freed, the test verifies that the first leader that was queued (LeaderC) is the next to be processed, not any submitted later — proving fairness is maintained.
  1. Run the test: 
    # This command specifically runs the priority dispatch test
go clean -testcache && go test -v -run ^TestPriorityDispatch .
  2. Analyze results: the detailed test output logs the exact sequence of events. A successful run confirms the system's core scheduling and fairness guarantees are met under pressure.

Queue management testing

The queue management system implements sophisticated eviction and requeuing logic that requires specific testing to validate FIFO integrity and eviction behaviour under load.

9. Queue eviction logic (requeue_eviction_test.go)

These integration tests validate the queue's behaviour when full and the eviction logic when requeuing older requests.

The tests cover:

  • Simple queue behaviour: basic queue filling, rejection when full, and direct eviction testing.
  • FIFO preservation: ensures that when an older request is requeued into a full queue, the newest item is evicted while maintaining chronological order.
  • Queue state validation: verifies that queue status reporting accurately reflects the internal state after operations.
  1. Run the eviction tests: 
    # Run the specific queue eviction tests
go clean -testcache && go test -v -run ^TestSimpleQueueBehavior .
go clean -testcache && go test -v -run ^TestRequeueEvictionDirect .
  2. Analyze results: the tests log detailed queue states before and after operations. Successful runs demonstrate that FIFO order is preserved even when eviction occurs.
10. Queue manager unit logic (chatbot/queue/manager_test.go)

These unit tests validate the core queue manager functionality in isolation, testing the mathematical correctness of queue operations without external dependencies.

The tests validate:

  • Basic operations: enqueue, dequeue, and status reporting.
  • Capacity management: proper rejection when the queue is full.
  • FIFO eviction: complex eviction scenarios with multiple requeues.
  • Chronological insertion: ensures requeued requests are inserted at the correct position to maintain timestamp ordering.
  • Integration compliance: verifies the queue manager works correctly with the complete request structure.
  1. Run the queue manager unit tests: 
    # Run all queue manager tests
go clean -testcache && go test -v ./chatbot/queue
  2. Analyze results: these tests verify the mathematical correctness of queue operations. Focus on the chronological ordering validation in complex eviction scenarios.
